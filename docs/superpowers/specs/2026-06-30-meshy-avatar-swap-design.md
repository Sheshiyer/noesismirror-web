# Custom Meshy AI avatar — replacing the astronaut

## Context

The current scene mounts a rigged astronaut character (`public/models/Astronaut.glb` + 4 separate locomotion clips) as the player avatar. We want to swap it for a custom-designed humanoid avatar produced through Meshy AI's full image-to-3D + rigging + animation pipeline. The goal is a one-time bake (not per-user runtime generation): one curated avatar replaces the astronaut for everyone, while preserving the existing animation blend system, footstep audio, and terrain-following shader so nothing else in the scene changes.

The Meshy API key lives in a global `~/.claude/.env` file outside this repo. The Cloudflare Workers backend is **not** involved — this is an offline asset pipeline that produces files committed to `public/models/avatar/`.

## Decisions locked in advance

| # | Decision | Source |

## Pipeline overview

```
[1] Concept image                  [2] Meshy image-to-3D       [3] Meshy rigging                [4] Meshy animations             [5] Code swap
    generate_image (Higgsfield)    POST /image-to-3d           POST /rigging                    POST /animations × 2             character/config.ts
    A-pose humanoid PNG            pose_mode: "a-pose"         input_task_id from step 2        action_id 0   → Idle.glb        flip MODEL_PATHS to
    public/concepts/source.png     enable_pbr: true            returns rigged GLB               action_id 544 → WalkingBack.glb  /models/avatar/*.glb
                                   target_polycount 30000      + Walking + Running (free)       (Walking + Running already       drop external KTX2
                                                                                                  baked by step 3)               material override
```

All steps 1–4 run inside one CLI script (`scripts/bake-avatar.mjs`). Step 5 is a small focused edit to two existing files. No new components, no new routes, no Workers code.

## Meshy API contract (verified from docs)

**Auth:** `Authorization: Bearer ${MESHY_API_KEY}` on every call.

**Image-to-3D** — `POST /openapi/v1/image-to-3d`
```jsonc
{
  "image_url": "data:image/png;base64,...",   // base64 data URI of source
  "ai_model": "meshy-6",
  "should_texture": true,
  "enable_pbr": true,
  "target_polycount": 30000,
  "topology": "triangle",
  "target_formats": ["glb"],
  "pose_mode": "a-pose"                        // CRITICAL — rigging API requires A-pose
}
```
Poll `GET /openapi/v1/image-to-3d/{id}` until `status === "SUCCEEDED"`. Save `id` as `image_task_id` for step 3.

**Rigging** — `POST /openapi/v1/rigging`
```jsonc
{
  "input_task_id": "<image_task_id>",   // chains to the image-to-3D output
  "height_meters": 1.7                   // standard humanoid; matches existing astronaut scale
}
```
Poll `GET /openapi/v1/rigging/{id}` until `SUCCEEDED`. Download three files from the response:
- `result.rigged_character_glb_url` → `public/models/avatar/Avatar.glb`
- `result.basic_animations.walking_glb_url` → `public/models/avatar/Walking.glb`
- `result.basic_animations.running_glb_url` → `public/models/avatar/Running.glb`

Save `id` as `rig_task_id` for step 4.

**Animations** — `POST /openapi/v1/animations` (called twice)
```jsonc
{ "rig_task_id": "<rig_task_id>", "action_id": 0   }   // Idle
{ "rig_task_id": "<rig_task_id>", "action_id": 544 }   // Walk_Backward
```
Poll `GET /openapi/v1/animations/{id}` each until `SUCCEEDED`. Download `result.animation_glb_url`:
- action 0   → `public/models/avatar/Idle.glb`
- action 544 → `public/models/avatar/WalkingBack.glb`

**Costs (estimate):** ~30 (image-to-3D) + 10 (PBR textures) + ~50 (rigging) + 3 × 2 (animations) ≈ **~100 credits per bake**. URLs expire in ~3 days — script downloads eagerly.

## Bake script — `scripts/bake-avatar.mjs`

Single Node 20+ file, no dependencies beyond built-ins. Flow:

1. Parse `~/.claude/.env` for `MESHY_API_KEY` using the same regex pattern `scripts/admin.ts:34-50` already uses for `.env`.
2. Parse CLI args: `--image <path>`, `--out public/models/avatar/`, optional `--idle-id 0`, `--back-id 544`, `--height 1.7`, `--resume <manifest-path>` (to skip steps already completed if a previous run crashed mid-pipeline).
3. Read `--image` from disk, base64-encode, build `data:image/png;base64,...` URI.
4. Call image-to-3D, poll every 5 s, print `progress %` on each tick.
5. Call rigging with the resulting `input_task_id`, poll, download the three GLBs.
6. Call animations × 2 with the resulting `rig_task_id`, poll each, download the two GLBs.
7. Write `public/models/avatar/bake-manifest.json` containing: source image SHA-256, all 3 task IDs, credits consumed per task, total wall clock, action_ids used, timestamps. This makes re-baking and auditing trivial.
8. Print a one-line next-step summary: "Set `useCustomAvatar = true` in `character/config.ts` and run `npm run dev`."

Error handling style mirrors `scripts/admin.ts`: check `response.ok`, parse JSON with fallback to text, throw with a descriptive message that includes the Meshy task ID for support tickets. On `FAILED`/`CANCELED`, print `task_error.message` verbatim.

## Code swap

### `src/components/character/config.ts`

Add a feature flag and an "avatar profile" abstraction so the swap is one line to flip:

```ts
export const AVATAR_PROFILE: 'astronaut' | 'custom' = 'custom';

const PROFILES = {
  astronaut: {
    modelPaths: ['/models/Astronaut.glb', '/models/Idle.glb', '/models/Walking.glb', '/models/Running.glb', '/models/WalkingBack.glb'],
    bodyMeshNames: ['Astronaut_Suit_Body_Detail_01_Mesh', 'Astronaut_Suit_Body_Mesh', 'Astronaut_Suit_Shoes_Mesh'],
    helmetMatch: 'Helmet',
    hideMatch: 'Person',
    useExternalMaterials: true,
    bodyTexturePaths: BODY_TEXTURE_PATHS,
    detailTexturePaths: DETAIL_TEXTURE_PATHS,
  },
  custom: {
    modelPaths: ['/models/avatar/Avatar.glb', '/models/avatar/Idle.glb', '/models/avatar/Walking.glb', '/models/avatar/Running.glb', '/models/avatar/WalkingBack.glb'],
    bodyMeshNames: [],                  // filled in after first bake by inspecting the GLB
    helmetMatch: null,
    hideMatch: null,
    useExternalMaterials: false,         // Meshy bakes PBR into the GLTF — keep its materials
    bodyTexturePaths: null,
    detailTexturePaths: null,
  },
} as const;

export const ACTIVE_PROFILE = PROFILES[AVATAR_PROFILE];
```

### `src/components/character/hooks/useCharacterAssets.ts`

Wrap the existing material-override block (currently lines 81–97) in `if (ACTIVE_PROFILE.useExternalMaterials)`. When false, leave the GLTF's bundled materials alone — Meshy already baked PBR maps into the GLB, so we save 21 MB of KTX2 traffic and several traverse passes.

The `useGLTF`/`SkeletonUtils.clone` flow at lines 33–44 stays untouched. The bone hierarchy is industry-standard humanoid (Meshy outputs Unity-Humanoid / UE-Mannequin compatible rigs), so `useAnimations(animations, sceneRef)` in [useCharacterPhysics.ts:33](src/components/character/hooks/useCharacterPhysics.ts:33) will resolve `actions['Idle']`, `actions['Walk']`, `actions['Run']`, `actions['Back']` from the clip names — provided the bake script renames the clips inside each downloaded GLB to those exact strings (Meshy returns them under generic names; rename happens in step 6/7 of the bake script using a tiny three.js post-process or by setting `clip.name` on save).

### `src/components/WorldController.tsx`

No change. `<Character position={[0,0,0]} scale={1} visible={enableCharacter} />` works identically.

## Out of scope

- Per-user avatar generation, runtime Meshy calls, upload UI, R2 caching of user avatars — all explicitly rejected by user.
- Worker-side `/api/meshy/*` routes — not needed for one-shot bake.
- Mixamo / AccuRig / Blender steps — Meshy handles rigging natively, makes manual tools redundant.
- Re-tuning the contact shadow / camera offsets — leave defaults; if the new avatar's silhouette demands a tweak, treat as follow-up.
